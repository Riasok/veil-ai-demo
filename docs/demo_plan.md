## 1. 문제,  은행 내부 데이터는 LLM으로 줄줄 샌다

은행 직원이 ChatGPT·Claude에 업무를 시키는 순간, 아래 데이터가 프롬프트에 그대로 들어감.

**은행 안에서 오가는 민감 데이터**

- 고객 식별정보: 이름, 주민번호, 연락처, 주소
- 금융 거래정보: 계좌번호, 잔액, 대출·카드 내역, 금리
- 신용 정보: 연체 이력, 신용등급, 소득, 재무제표
- 기술 자산: 소스코드, DB 접속정보, API 키, 내부 시스템 구조

이게 외부 LLM으로 나가면 **개인정보보호법·신용정보법 위반 + 영업비밀 유출**.

## 2. 데모 시나리오 3종 , 누가 / 무엇을 / 어떻게 흘리는가

직원 유형 × 데이터 형태로 대표 상황 3개를 구성. (※ PoC 단계에서 시나리오는 얼마든지 추가 가능)

| 시나리오 | 직무 | 데이터 형태 | 유출 특성 |
| --- | --- | --- | --- |
| **민원 응대** | 창구·상담 직원 | 비정형 텍스트 | 한 건이지만 연체 사실 등 맥락까지 샘 |
| **고객 리스트 분석** | 분석가·기획자 | 정형 표(CSV) | 한 번에 수백~수천 건, 임팩트 최대 |
| **개발 디버깅** | 개발자 | 로그·코드·설정 | "데이터"로 인식 못 해 경계심↓ + 시크릿 혼입 |

각 시나리오 안의 데이터 항목은 **법령(신용정보법 §2)에서**, 민원 화면 구조는 **실제 판례에서** 가져와 현실성 확보. (→ 3, 4번)

## 3. 핵심 메시지,  왜 룰베이스(정규식)만으로는 안 되는가

**이게 발표의 중심.** 명백한 패턴은 누구나 잡는다. 진짜 어려운 건 "사람도 헷갈리는 회색지대"이고, 여기가 우리 제품의 차별화 지점.

### ① 결합 식별 — 단독으론 안 걸리는데 합치면 식별됨

법(신용정보법 §2 2호)이 *"그 자체로는 식별 불가여도 다른 정보와 쉽게 결합하면 식별되는 것"*도 개인신용정보로 규정.

> 예: "서초구 + 1973년생 + 정금속공업 대표이사" → 이름·주민번호 없지만 **사실상 1명으로 특정됨.**
정규식은 이름도 번호도 없으니 **그냥 통과시킨다.** → 우리는 결합 위험을 점수화해 차단.
> 

### ② 맥락 판단 — 같은 값이 상황 따라 다름

> "이체 한도를 5,000만원으로" (요청 → 통과) vs "잔액 5,127만원" (거래정보 → 차단)
같은 금액인데 의미가 반대. **정규식은 둘 다 잡거나 둘 다 놓친다.**
> 

### ③ 회피 시도 — 변형하면 정규식이 뚫림

> "731204-1773971" 은 잡아도 → "칠삼일이공사 1773971"(한글), "731204 1773971"(공백 삽입)은?
**정규식 우회. 우리는 정규화 후 의미로 탐지.**
> 

### ④ 기술자산 경계 — 개인정보 아닌데 막아야 함

> 변수명 `rrn`(코드 구조 → 통과) vs 그 변수의 실제 값(런타임 → 차단).
법령 분류엔 없지만 내부 시스템 구조·시크릿은 영업비밀. **맥락으로 구분 필요.**
> 

**정리: 탐지(찾기)는 룰베이스로 되지만, "이게 진짜 위험한가"의 판단은 AI가 해야 한다.**

## 4. 법적 근거,  탐지 결과마다 조문을 댈 수 있다

감독당국·정보보호팀이 쓰는 언어로 라벨링. "심사 통과"를 셀링하려면 필수.

| 데이터 | 법적 분류 | 근거 | 비고 |
| --- | --- | --- | --- |
| 이름·연락처·주소 | 식별정보 | 신용정보법 §2-1의2 |  |
| 계좌·잔액·카드·금리 | 거래내용정보 | 신용정보법 §2-1의3 |  |
| 연체·금융사기 | 신용도판단정보 | 신용정보법 §2-1의4 |  |
| 소득·재무제표 | 신용거래능력정보 | 신용정보법 §2-1의5 |  |
| 신용등급·평점 | 기타 신용판단정보 | 신용정보법 §2-1의6 |  |
| **주민번호** | 고유식별정보 | 개인정보보호법 §24의2, 시행령 §19 | **법령 근거 없으면 처리 금지 → 유일하게 BLOCK** |
| 비밀번호·OTP·인증서 | 접근매체 | 전자금융거래법 §2 | 시크릿 탐지 근거 |

**가장 중요한 조항 — 신용정보법 §2 2호**: "다른 정보와 쉽게 결합하여 식별 가능한 것"도 보호 대상.
→ 이게 **3번 ①결합식별의 법적 근거**. 단순 마스킹으론 법 요구를 못 맞춘다는 우리 기술의 정당화.

**판례로 보강 (민원 시나리오)**: 서울중앙지법 2021가합574498 — 은행 단말의 *종합고객정보조회/거래내역조회 화면 필드(실명번호·계좌번호·잔액 등)*가 개인신용정보라고 법원이 판단. → 직원이 화면 복사해 붙여넣는 경로가 실제 위법임을 입증.

## 5. 제품이 하는 일, 3단계

| 단계 | 하는 일 | 누가 정하나 |
| --- | --- | --- |
| **1. 탐지** | 민감 항목 찾기 (정규식+체크섬 / **+ 회색지대 AI 판단**) | 제품 — 여기가 차별화 |
| **2. 분류** | 신용정보법 §몇 조 무슨 정보인지 라벨링 | 법령이 규정 |
| **3. 정책** | 차단·가명화·통과 결정 | 은행 정책 + 제품이 함께 |

법은 "무엇이 민감한가(1·2)"까지만 정함. **"어떻게 막을까(3)"는 정답이 없어 고객사와 맞춤.** 제품 가치는 1단계(회색지대 탐지)와 3단계(정책 유연성)에서 나온다.
